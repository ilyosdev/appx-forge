# File Push Protocol

Version: 0.1.0
Last updated: 2026-04-15

## Overview

File push allows SDK/CLI clients to send files directly to the agent hosting a sandbox, avoiding the control plane as a bandwidth bottleneck. The control plane acts as an authenticator and router -- it validates the request, looks up which agent hosts the sandbox, and returns a signed redirect URL pointing directly at the agent.

This design means file payloads (which can be large during AI code generation) never flow through the control plane. The control plane handles only the lightweight redirect, and the agent handles the actual file I/O.

## Flow

```
SDK/CLI                     Control Plane                    Agent
  |                              |                              |
  |  POST /v1/sandboxes/{id}/files                              |
  |  Authorization: Bearer <api_key>                            |
  |  Content-Type: application/json                             |
  |  Body: { files: [...] }      |                              |
  |----------------------------->|                              |
  |                              |  1. Validate auth            |
  |                              |  2. Look up sandbox -> node  |
  |                              |  3. Generate HMAC-SHA256     |
  |                              |     signed URL               |
  |                              |                              |
  |  307 Temporary Redirect      |                              |
  |  Location: http://{ts_ip}:{port}/v1/sandboxes/{id}/files    |
  |            ?expires={unix_ts}&sig={hex_hmac}                |
  |<-----------------------------|                              |
  |                              |                              |
  |  POST http://{ts_ip}:{port}/v1/sandboxes/{id}/files         |
  |       ?expires={unix_ts}&sig={hex_hmac}                     |
  |  (same body as original request)                            |
  |------------------------------------------------------------->|
  |                              |  4. Validate signature       |
  |                              |  5. Check expiry             |
  |                              |  6. Write files to bind-mount|
  |                              |                              |
  |  200 OK                      |                              |
  |  { "written": [...], "failed": [] }                         |
  |<-------------------------------------------------------------|
```

### Step-by-Step

1. **Client** sends `POST /v1/sandboxes/{id}/files` to the control plane with the file payload and a valid API key or session token.

2. **Control plane** authenticates the request, looks up the sandbox's `node_id`, and retrieves the node's `tailscale_ip` and `agent_listen_port`.

3. **Control plane** generates an HMAC-SHA256 signed URL targeting the agent, with a 60-second expiry.

4. **Control plane** returns `307 Temporary Redirect` with `Location` header set to the signed URL.

5. **Client** follows the redirect, sending the same request body to the agent's direct endpoint.

6. **Agent** validates the HMAC signature and checks that the URL has not expired.

7. **Agent** writes the files to the sandbox's bind-mount directory: `/var/lib/forge/sandboxes/{app_name}/code/`.

8. **Agent** returns `200 OK` with a summary of written and failed files.

## Request Format

Both the control plane and agent accept two content types:

### JSON Format

```http
POST /v1/sandboxes/{id}/files
Content-Type: application/json

{
  "files": [
    {
      "path": "App.tsx",
      "content": "aW1wb3J0IFJlYWN0IGZyb20gJ3JlYWN0Jzs=",
      "delete": false
    },
    {
      "path": "screens/Home.tsx",
      "content": "ZXhwb3J0IGRlZmF1bHQgZnVuY3Rpb24gSG9tZSgpIHt9",
      "delete": false
    },
    {
      "path": "screens/OldScreen.tsx",
      "content": "",
      "delete": true
    }
  ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `files` | array | yes | List of file operations |
| `files[].path` | string | yes | Relative path within the code directory (no leading `/`) |
| `files[].content` | string | yes (unless delete) | Base64-encoded file content |
| `files[].delete` | boolean | no (default: false) | If true, delete the file instead of writing |

Path traversal prevention: the agent rejects any path containing `..` or starting with `/`.

### Tar Format

```http
POST /v1/sandboxes/{id}/files
Content-Type: application/x-tar

<binary tar.gz data>
```

The tar archive is extracted directly to the code directory. Symlinks in the archive are ignored for security.

## Signed URL Format

The signed URL generated by the control plane has this structure:

```
http://{tailscale_ip}:{agent_listen_port}/v1/sandboxes/{sandbox_id}/files?expires={unix_ts}&sig={hex_hmac}
```

| Parameter | Description |
|-----------|-------------|
| `tailscale_ip` | The Tailscale IP of the node hosting the sandbox |
| `agent_listen_port` | The agent's HTTP listen port (default: 8090) |
| `sandbox_id` | UUID of the target sandbox |
| `expires` | Unix timestamp, 60 seconds from URL generation time |
| `sig` | HMAC-SHA256 signature as hex string |

### Signature Computation

The HMAC-SHA256 signature is computed over a canonical string:

```
message = "/v1/sandboxes/{sandbox_id}/files?sandbox_id={sandbox_id}&expires={unix_ts}"
sig     = hex(HMAC-SHA256(shared_secret, message))
```

- The `shared_secret` is a pre-shared key known to both the control plane and all agents (distributed via environment variable `FORGE_HMAC_SECRET`).
- The message uses the full URL path and includes the sandbox_id and expires parameters in a canonical order.
- Both the control plane and agent use the same key and the same canonical string format.

### Signature Verification (Agent Side)

```go
func verifySignature(path, sandboxID, expiresStr, providedSig string, secret []byte) error {
    // 1. Check expiry
    expires, err := strconv.ParseInt(expiresStr, 10, 64)
    if err != nil {
        return fmt.Errorf("parsing expires: %w", err)
    }
    if time.Now().Unix() > expires {
        return errors.New("signed URL expired")
    }

    // 2. Compute expected signature
    message := fmt.Sprintf("%s?sandbox_id=%s&expires=%s", path, sandboxID, expiresStr)
    mac := hmac.New(sha256.New, secret)
    mac.Write([]byte(message))
    expectedSig := hex.EncodeToString(mac.Sum(nil))

    // 3. Constant-time comparison
    if !hmac.Equal([]byte(expectedSig), []byte(providedSig)) {
        return errors.New("invalid signature")
    }

    return nil
}
```

Key points:
- **Always check expiry before computing HMAC** to reject expired URLs cheaply.
- **Use `hmac.Equal()` for constant-time comparison** to prevent timing attacks.
- **60 seconds** is the maximum URL lifetime -- long enough for large file uploads over Tailscale, short enough to limit replay risk.

## Response Format

### Success (200 OK)

```json
{
  "written": ["App.tsx", "screens/Home.tsx"],
  "failed": []
}
```

### Partial Failure (200 OK with failed entries)

```json
{
  "written": ["App.tsx"],
  "failed": ["screens/Home.tsx"]
}
```

A partial failure still returns 200. The caller should inspect the `failed` array and retry if needed.

## Error Handling

| Scenario | HTTP Status | Response |
|----------|-------------|----------|
| Agent unreachable after redirect | Client-side error | Client retries the 307 redirect once, then fails |
| Signature invalid | 403 Forbidden | `{ "error": "invalid signature" }` |
| URL expired | 403 Forbidden | `{ "error": "signed URL expired" }` |
| Sandbox not on this agent | 404 Not Found | `{ "error": "sandbox not found on this node" }` |
| Path traversal attempt | 400 Bad Request | `{ "error": "invalid file path: must not contain .." }` |
| Write failure (disk full) | 500 Internal Server Error | `{ "error": "write failed", "failed": ["App.tsx"] }` |
| Write failure (permission) | 500 Internal Server Error | `{ "error": "permission denied", "failed": ["App.tsx"] }` |

## Client Behavior

### Redirect Handling

Most HTTP clients follow 307 redirects automatically, preserving the request method and body. However:

- The client **must** ensure the HTTP library follows 307 with the same method (POST) and body. Some libraries downgrade to GET on redirect.
- If the redirect target is unreachable, the client retries the original request to the control plane once. If the control plane returns another 307 to the same unreachable agent, the client fails with an error.
- The client does **not** need to add any authorization header when following the redirect -- the signed URL query parameters serve as authentication.

### Content-Type Preservation

The client must send the same `Content-Type` header to both the control plane and the agent (via redirect). The control plane does not inspect the body; it only uses the sandbox ID from the URL path.

## Security Considerations

- **HMAC shared secret rotation**: The control plane and agents must use the same `FORGE_HMAC_SECRET`. To rotate, set the new secret on all agents first, then on the control plane. During rotation, agents should accept signatures from both the old and new secret (dual-key validation window).
- **Tailscale mesh**: The agent HTTP server listens only on the Tailscale IP. File push traffic never traverses the public internet. Even if a signed URL leaks, it is only valid for 60 seconds and only reachable within the Tailscale mesh.
- **No credential forwarding**: The client's API key is validated by the control plane only. The agent authenticates via the HMAC signature on the URL -- no API keys or tokens are sent to the agent.
