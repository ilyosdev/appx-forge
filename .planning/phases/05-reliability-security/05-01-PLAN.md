<objective>
TDD implementation of auto-restart with exponential backoff for crashed sandbox containers.

Purpose: When a running sandbox's container exits unexpectedly, the system must automatically attempt restart up to 3 times with exponential backoff (5s, 10s, 20s). After 3 failures, the sandbox transitions to FAILED. This prevents transient crashes from killing sandboxes while avoiding infinite restart loops.

```go
package lifecycle

// RestartStore abstracts DB operations for restart logic.
type RestartStore interface {
    IncrementSandboxFailureCount(ctx context.Context, id pgtype.UUID) (store.Sandbox, error)
    ResetSandboxFailureCount(ctx context.Context, id pgtype.UUID) error
    TransitionSandboxState(ctx context.Context, arg store.TransitionSandboxStateParams) (store.Sandbox, error)
    CreateCommand(ctx context.Context, arg store.CreateCommandParams) (store.Command, error)
    RecordEvent(ctx context.Context, arg store.RecordEventParams) (store.Event, error)
    GetSandbox(ctx context.Context, id pgtype.UUID) (store.Sandbox, error)
}

const maxRestartAttempts = 3

// RestartManager handles crash recovery with exponential backoff.
type RestartManager struct {
    store  RestartStore
    logger *slog.Logger
}
```

- `HandleCrash(ctx, sandbox store.Sandbox)`: increments failure_count, if count < maxRestartAttempts computes delay (5s * 2^count), schedules restart_sandbox command, transitions restarting->starting. If count >= maxRestartAttempts, transitions to failed.
- `HandleRestarted(ctx, sandboxID pgtype.UUID)`: resets failure_count to 0.
- Backoff formula: `baseDelay * (1 << failureCount)` where baseDelay = 5s.
- NOTE: The actual delay is recorded as command timeout metadata -- the agent handles the delay. The control plane dispatches the restart command immediately but the agent uses the delay before executing.

Run `go test ./control/internal/lifecycle/ -run TestRestart -count=1 -race` to confirm GREEN.
