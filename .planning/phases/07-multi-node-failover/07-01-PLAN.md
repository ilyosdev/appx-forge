   ```go
   type RescheduleStore interface {
       ListRunningSandboxesByNode(ctx context.Context, nodeID pgtype.UUID) ([]store.Sandbox, error)
       ListHealthyNodes(ctx context.Context) ([]store.Node, error)
       TransitionSandboxState(ctx context.Context, arg store.TransitionSandboxStateParams) (store.Sandbox, error)
       AssignSandboxToNode(ctx context.Context, arg store.AssignSandboxToNodeParams) (store.Sandbox, error)
       CreateCommand(ctx context.Context, arg store.CreateCommandParams) (store.Command, error)
       RecordEvent(ctx context.Context, arg store.RecordEventParams) (store.Event, error)
       ResetSandboxFailureCount(ctx context.Context, id pgtype.UUID) error
   }
   ```

2. **RescheduleResult struct**:
   ```go
   type RescheduleResult struct {
       Count  int      // Number of sandboxes successfully rescheduled
       Failed int      // Number of sandboxes that failed to reschedule
       Errors []error  // Collected errors from failed reschedules
   }
   ```

3. **Rescheduler struct** with `store RescheduleStore`, `notifier RouteNotifier`, `logger *slog.Logger`

4. **NewRescheduler(store, notifier, logger)** constructor (nil logger defaults to slog.Default)

5. **RescheduleNode(ctx, failedNodeID pgtype.UUID) (RescheduleResult, error)** method:
   - Call `store.ListRunningSandboxesByNode(ctx, failedNodeID)` to get RUNNING sandboxes
   - If empty, return `RescheduleResult{Count: 0}` immediately
   - For each sandbox, call `rescheduleSandbox(ctx, sandbox)` and collect results
   - Return aggregate result with count/failed/errors

6. **rescheduleSandbox(ctx, sandbox store.Sandbox) error** private method:
   a) Transition `running -> pending` via EventNodeFailed:
      ```go
      store.TransitionSandboxState(ctx, store.TransitionSandboxStateParams{
          State:   string(models.StatePending),
          ID:      sandbox.ID,
          State_2: string(models.StateRunning),
      })
      ```
   b) Record `node_failed` event with actor "rescheduler"
   c) Remove Caddy route via `notifier.OnSandboxStopped(ctx, sandbox.AppName)` (best-effort, log warning on error)
   d) Reset failure count to 0 via `store.ResetSandboxFailureCount` (fresh start on new node)
   e) List healthy nodes, convert to `scheduler.NodeCandidate` slice
   f) Parse Resources JSON from sandbox to get memory_mb (default 512 if missing)
   g) Call `scheduler.Schedule(candidates, requiredMB)` to pick new node
   h) If scheduler returns ErrNoCapacity or ErrNoNodes: transition sandbox to FAILED, record event, return error
   i) Assign sandbox to new node via `store.AssignSandboxToNode` (transitions pending -> starting)
   j) Create `start_sandbox` command targeting the new node with payload containing app_name, image, resources, env
   k) Record `scheduled` event with actor "rescheduler"
   l) Log: `logger.Info("sandbox rescheduled", "sandbox_id", ..., "from_node", ..., "to_node", ...)`

Key design points:
- Sequential processing per sandbox (not concurrent) -- same pattern as IdleReaper.reap()
- Best-effort route removal (errors logged, never propagated)
- Partial failure tolerance: one sandbox failing does not abort the batch
- No retry logic -- if rescheduling fails, the sandbox ends up in FAILED and the operator can manually intervene

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| heartbeat monitor -> rescheduler | Internal call within control plane; no untrusted input |
| rescheduler -> scheduler | Pure function call; node candidates from trusted DB |
| rescheduler -> Caddy Admin API | Route removal; best-effort, errors non-fatal |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | D (DoS) | RescheduleNode | mitigate | Sequential per-sandbox processing prevents thundering herd; no concurrent reschedules |
| T-07-02 | T (Tampering) | State transition | mitigate | CAS (UPDATE WHERE state=$expected) prevents double-reschedule of same sandbox |
| T-07-03 | I (Info Disclosure) | Logging | mitigate | Log sandbox_id and node_id only, no env vars or secrets in reschedule logs |
| T-07-04 | D (DoS) | Cascade reschedule | mitigate | If rescheduled sandbox fails immediately on new node, restart manager handles it (max 3 retries), not rescheduler |
</threat_model>
