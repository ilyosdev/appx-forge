1. Create `docs/contracts/agent-protocol.md` with the following sections:

   ## Overview
   Forge agents are stateless binaries running as systemd services on each node. They communicate with the control plane via HTTP.

   ## Registration
   - On startup, agent sends `POST /v1/nodes/register` with:
     ```json
     {
       "hostname": "node-1",
       "tailscale_ip": "100.64.1.5",
       "agent_listen_port": 8090,
       "capacity_mb": 24000,
       "capacity_cpu": 8.0,
       "agent_version": "0.1.0"
     }
     ```
   - Control plane returns: `{ "node_id": "uuid", "agent_token": "secret", "heartbeat_interval_seconds": 15 }`
   - Agent stores `node_id` and `agent_token` in memory (not on disk).
   - `agent_token` is included as `Authorization: Bearer <agent_token>` on all subsequent requests.
   - Re-registration with same `hostname` + `tailscale_ip` returns existing `node_id` (idempotent).

   ## Heartbeat
   - Every `heartbeat_interval_seconds` (default 15s), agent sends `POST /v1/nodes/{id}/heartbeat` with:
     ```json
     { "used_mb": 8500, "running_containers": 12 }
     ```
   - Control plane updates `last_seen_at`, `used_mb`, `running_containers`.
   - 3 consecutive missed heartbeats (45s) marks node as `unhealthy`.

   ## Command Polling (Long-Poll)
   - Agent calls `GET /v1/agents/{id}/commands?wait=30` continuously.
   - Control plane holds the connection for up to `wait` seconds (default 30, max 60).
   - Returns `{ "commands": [...] }` -- array of 0+ commands.
   - If no commands pending, returns `{ "commands": [] }` after timeout.
   - Agent processes each command and acks.

   ## Command Types
   Document each command type with its payload schema:
   - `start_sandbox`: `{ "sandbox_id": "uuid", "app_name": "...", "image": "...", "resources": { "cpu_cores": 0.5, "memory_mb": 512 }, "env": { "APP_NAME": "..." } }`
   - `stop_sandbox`: `{ "sandbox_id": "uuid", "container_id": "..." }`
   - `restart_sandbox`: `{ "sandbox_id": "uuid", "container_id": "..." }`
   - `get_logs`: `{ "sandbox_id": "uuid", "container_id": "...", "tail": 100, "follow": false }`
   - `prune`: `{}` (remove stopped containers and unused images)

   ## Command Acknowledgment
   - After executing a command, agent sends `POST /v1/agents/{id}/commands/{cmd_id}/ack` with:
     ```json
     { "status": "success", "result": { "container_id": "abc123", "host_port": 43210 } }
     ```
   - Or on failure: `{ "status": "failure", "error": "image not found: appx/sandbox:v99" }`
   - Ack is idempotent -- sending it twice is safe.

   ## Event Reporting
   - Agent watches Docker events stream (`type=container, event=die|oom`).
   - On container event, sends `POST /v1/agents/{id}/events` with:
     ```json
     {
       "sandbox_id": "uuid",
       "event_type": "container_exited",
       "container_id": "abc123",
       "exit_code": 137,
       "payload": { "oom_killed": true }
     }
     ```
   - Event types: `container_started`, `container_exited`, `container_oom`, `container_unhealthy`
   - Events are fire-and-forget (agent retries once on network error, then drops).

   ## Reconnection
   - Docker events stream: on disconnect, reconnect with `Since` timestamp of last received event. No events missed.
   - Control plane connection: on HTTP error, exponential backoff (1s, 2s, 4s, 8s, max 30s). Reset backoff on success.

2. Create `docs/contracts/filepush-protocol.md` with:

   ## Overview
   File push allows SDK/CLI clients to send files directly to the agent hosting a sandbox, avoiding control plane as a bandwidth bottleneck.

   ## Flow
   1. Client sends `POST /v1/sandboxes/{id}/files` to control plane with file payload.
   2. Control plane looks up sandbox's `node_id` and `tailscale_ip`.
   3. Control plane generates HMAC-SHA256 signed URL targeting agent: `http://{tailscale_ip}:{agent_listen_port}/v1/sandboxes/{id}/files?expires={unix_ts}&sig={hex_hmac}`
   4. Control plane returns `307 Temporary Redirect` with `Location: {signed_url}`.
   5. Client follows redirect, sending same body to agent.
   6. Agent validates signature (using shared HMAC key), checks expiry (60s).
   7. Agent writes files to bind-mount directory: `/var/lib/forge/sandboxes/{app_name}/code/`.
   8. Agent returns `200 OK` with `{ "written": ["App.tsx", "package.json"], "failed": [] }`.

   ## Request Format
   Two formats accepted by both control plane and agent:
   - `application/json`: `{ "files": [{ "path": "App.tsx", "content": "base64...", "delete": false }] }`
   - `application/x-tar`: tar.gz archive extracted to code directory

   ## Signed URL Format
   - URL: `http://{tailscale_ip}:{port}/v1/sandboxes/{sandbox_id}/files?expires={unix_ts}&sig={hex}`
   - `expires`: Unix timestamp, 60 seconds from issue time
   - `sig`: HMAC-SHA256 of `{path}?sandbox_id={id}&expires={ts}` using shared secret
   - Verification: agent recomputes HMAC with same key, uses `hmac.Equal()` for constant-time comparison
   - Expired or invalid signature returns `403 Forbidden`

   ## Error Handling
   - Agent unreachable: client retries 307 redirect once, then fails
   - Signature invalid: `403 Forbidden` with `{ "error": "invalid signature" }`
   - Sandbox not on this agent: `404 Not Found`
   - Write failure (disk full, permission): `500 Internal Server Error` with failed file list

3. Create `docs/contracts/proxy-routing.md` with:

   ## Overview
   The control plane manages Caddy reverse proxy routes via the Caddy Admin API. Each running sandbox gets a route mapping `{app_name}.myappx.live` to the node's Tailscale IP and host port.

   ## Caddy Base Config
   - Caddy runs on port 443 (TLS) and port 80 (redirect to HTTPS).
   - TLS via Cloudflare Origin CA certificate (15-year validity, no ACME/LE).
   - Admin API on `localhost:2019` (default, not exposed publicly).
   - WebSocket upgrade automatic (no extra config needed for Metro HMR).

   ## Route Addition
   When a sandbox reaches `running` state, control plane adds a route:
   ```
   POST http://localhost:2019/config/apps/http/servers/srv0/routes
   Content-Type: application/json

   {
     "@id": "route-{app_name}",
     "match": [{ "host": ["{app_name}.myappx.live"] }],
     "handle": [{
       "handler": "reverse_proxy",
       "upstreams": [{ "dial": "{tailscale_ip}:{host_port}" }],
       "transport": { "protocol": "http" },
       "headers": {
         "request": {
           "set": {
             "X-Forwarded-Host": ["{http.request.host}"],
             "X-Sandbox-ID": ["{sandbox_id}"]
           }
         }
       }
     }]
   }
   ```
   The `@id` field allows stable path-based access for updates and deletes.

   ## Route Removal
   When a sandbox is destroyed/stopped:
   ```
   DELETE http://localhost:2019/id/route-{app_name}
   ```
   Returns 200 on success, 404 if route already removed (idempotent).

   ## Batch Updates (WebSocket Drop Mitigation)
   - Caddy reloads config on every route change. Rapid changes (10+ sandboxes starting simultaneously) cause multiple reloads.
   - Each reload MAY drop in-flight WebSocket connections (Metro HMR).
   - Mitigation: control plane batches route changes with 500ms debounce.
   - Implementation: collect route changes in a buffer, flush every 500ms or when buffer reaches 50 changes.
   - Batch flush: `POST /load` with complete route config (not individual route additions).

   ## Drift Detection
   - Every 60 seconds, control plane queries:
     1. Caddy: `GET http://localhost:2019/config/apps/http/servers/srv0/routes` -- active routes
     2. Postgres: `SELECT app_name, tailscale_ip, host_port FROM sandboxes JOIN nodes ON ... WHERE state = 'running'` -- expected routes
   - Diff:
     - Route in Caddy but not in Postgres (orphan) -> DELETE from Caddy
     - Route in Postgres but not in Caddy (missing) -> POST to Caddy
     - Route in both but different upstream -> PUT update in Caddy
   - Drift detector only touches routes, NEVER containers. Routing drift and container state are separate concerns.
   - Log every correction as a warning: `slog.Warn("routing drift corrected", "app_name", appName, "action", "added|removed|updated")`

   ## Cloudflare DNS
   - Wildcard record: `*.myappx.live` CNAME or A record pointing to Caddy's public IP
   - Cloudflare proxy mode: ON (orange cloud) for DDoS protection
   - SSL mode: Full (Strict) with Origin CA cert on Caddy
   - HTTP/3 (QUIC): DISABLED in Cloudflare to avoid ERR_QUIC_PROTOCOL_ERROR
